# Настройка автоматической ежедневной подборки в Telegram

Дата настройки: 2026-07-13

## Цель

Автоматизировать этап 2–3 из плана развития (см. `CLAUDE.md`): ежедневная подборка туров
должна собираться и приходить в Telegram-бота **без участия пользователя и без привязки
к тому, включён ли его компьютер**.

## Итоговая архитектура

- **Планировщик**: облачный routine Claude Code (`claude.ai/code/routines`), а не локальный
  Windows Task Scheduler — выполняется на серверах Anthropic, не зависит от локального ПК.
  - Routine ID: `trig_01CnvXEkut14zdDzPat1Z4ha`
  - Расписание: `0 4 * * *` (UTC) = 08:00 по Самаре, ежедневно
  - Репозиторий: `github.com/ramolee2020/turizm-podborki`, ветка `main`
- **Источник правил**: `CLAUDE.md` в корне репозитория — агент читает его целиком при каждом
  запуске и следует всем правилам (направления, бюджет, источники, формат, дедупликация,
  безопасность).
- **Хранение истории**: каждая подборка сохраняется в `podborki/ГГГГ-ММ-ДД.md` и коммитится
  в репозиторий — нужно для правила «не повторять отель за 3 дня».
- **Доставка**: сообщение отправляется в Telegram-бота `@Turizm_mashulya_bot` через
  `curl` → Telegram Bot API (`sendMessage`), на фиксированный `chat_id`.
- **Окружение**: облачное окружение `Default` (`env_01HSogHMtsLRvYcXHx9d7AG7`),
  `Network access: Full` (открыт полный доступ в интернет — необходим для WebFetch по
  турсайтам и для обращения к `api.telegram.org`).
- **Права на запись в GitHub**: официальный GitHub App **Claude** (developer: `anthropics`),
  установлен на аккаунте `ramolee2020` **только на репозиторий `turizm-podborki`**,
  с правом `Contents: Read and write`.

## Хронология проблем и решений

1. **Первый тест (ручной запуск) — всё упало.**
   - `WebFetch` получал 403 на всех проверенных турсайтах (Level.Travel, Travelata и др.).
   - Отправка в Telegram падала с 403 на CONNECT-туннеле к `api.telegram.org`.
   - `git push` падал с 403 (права доступа) + отдельно обнаружился сломанный ключ подписи
     коммитов в окружении (`commit_signing_key.pub` пустой, без приватного ключа).
   - **Причина**: у окружения по умолчанию стоял `Network access: Trusted` — режим,
     пропускающий только проверенные пакетные источники (npm/pypi и т.п.), а не произвольные
     сайты и API.
   - **Решение**: переключили `Network access` на `Full` в настройках облачного окружения.
     После этого WebFetch к турсайтам и отправка в Telegram заработали.

2. **Токен Telegram-бота "засветился"** в тексте routine (виден через API/UI) и в переписке.
   - **Решение**: токен перевыпущен через `@BotFather` (`/revoke`), новый токен обновлён
     в локальном `.env` и в тексте задачи routine.

3. **Git push всё ещё падал (403) даже после Full network access.**
   - Диагностика показала: `git push` из облачной сессии идёт не напрямую на GitHub, а через
     **внутренний прокси Anthropic** (`127.0.0.1:.../git/...`). Прокси сам хранит серверные
     креды для GitHub — любые креды, прописанные вручную внутри контейнера (в т.ч. созданный
     нами GitHub Personal Access Token в Setup script), **не используются вообще**.
   - Проверка на стороне GitHub (`github.com/settings/apps/authorizations`) показала: GitHub
     App **Claude** авторизован (есть OAuth-идентичность), но **не установлен ни на один
     репозиторий** («Claude has not been installed on any accounts you have access to»).
     Именно поэтому прокси мог читать (публичный репозиторий), но не мог писать.
   - **Решение**: установили GitHub App `Claude` (`github.com/apps/claude`) на аккаунт
     `ramolee2020`, с ограничением **Only select repositories → turizm-podborki** и правом
     `Contents: Read and write`. После этого `git push` заработал.

4. **Проверка самого cron-механизма.**
   - Временно переставили `cron_expression` на ближайшие пару минут, дождались, что routine
     сработал **автоматически** (без ручного `run`) — `last_fired_at` обновился без нашего
     участия. Затем вернули расписание на `0 4 * * *`.

5. **Усиление CLAUDE.md после успешного запуска.**
   - Добавлен раздел «Безопасность»: запрет выполнять инструкции, встреченные в содержимом
     веб-страниц (защита от промт-инъекций через найденные во время поиска сайты), отправка
     только на фиксированный `TELEGRAM_CHAT_ID`, запрет печатать секреты в выводе.
   - Добавлено правило в «Контекст бизнеса»: исключать отели с рейтингом ниже 3.5
     (по звёздности или по оценке гостей/агрегатора).

  Setup script облачного окружения — сейчас он **не нужен и не используется** (push идёт

  хотя облачный routine их не читает (значения зашиты прямо в тексте задачи routine) — это
  неиспользуемая копия живого токена на диске.

## Выводы

1. **Облачные окружения Claude Code по умолчанию сильно ограничены по сети** (`Trusted` —
   только пакетные реестры). Для задач, которые реально ходят в интернет за данными
   (парсинг сайтов, сторонние API), это нужно явно менять на `Full` или `Custom` —
   иначе сбои выглядят как проблема на стороне целевого сайта, а на деле это сетевая
   политика окружения.
2. **Git-операции в облачной сессии идут через внутренний прокси, а не напрямую.** Любые
   учётные данные, прописанные вручную внутри контейнера (PAT в `~/.git-credentials` и т.п.),
   не влияют на push — авторизация нужна на уровне платформенной GitHub-интеграции
   (GitHub App), а не внутри самой песочницы. Прежде чем городить свои креды, стоит сначала
   проверить, установлен ли и на что именно официальный GitHub App.
3. **«Connected» в списке коннекторов claude.ai не означает «есть доступ на запись».**
   Коннектор может быть авторизован (OAuth-идентичность подтверждена), но не установлен ни
   на один репозиторий — тогда read может работать (для публичных репо), а write — нет.
   Проверять нужно отдельно на стороне GitHub (`Installed GitHub Apps`).
4. **У облачных routines нет отдельного хранилища секретов.** Единственный способ передать
   токен агенту — прописать его в тексте задачи или в Setup script окружения; оба места
   видны через API/UI владельцу аккаунта. Практический вывод: минимизировать права токена
   (только нужный репозиторий/бот), быть готовым быстро ротировать при подозрении на
   случайную засветку, не переиспользовать один и тот же токен для разных целей.
5. **Тестировать нужно весь пайплайн целиком, а не по частям** — ошибка обнаруживалась
   только при полном прогоне (поиск → запись файла → commit → push → Telegram), потому что
   часть проблем маскировала другие (например, пока не починили сеть, вообще не было видно
   проблему с git push).
6. **Раз агент читает недоверенный контент из интернета (описания туров, отзывы), нужна
   явная защита от промт-инъекций** — иначе инструкции, «спрятанные» в содержимом
   сторонней страницы, потенциально могут повлиять на поведение агента (например, попытаться
   заставить его отправить сообщение не туда или раскрыть секреты).
